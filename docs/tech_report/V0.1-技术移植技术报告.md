## 2. Lib 模块详解

### 2.1 `lib/env.ts` — 环境变量校验

**移植自**: `packages/engine/src/env.ts`

**变更**: agentset 使用 `@t3-oss/env-nextjs` + `zod/v4`，管理 Pinecone/Turbopuffer/Azure/Cohere 等多服务密钥（约 10 个变量）。我们简化为 4 个变量。

```typescript
// 变量表
OPENAI_API_KEY       // 必填，无默认值 — 启动即校验
OPENAI_BASE_URL      // 默认 "https://api.openai.com/v1"
LLM_MODEL            // 默认 "gpt-4.1"
EMBEDDING_MODEL      // 默认 "text-embedding-3-small"
```

**缺省策略**: `emptyStringAsUndefined: true` — 空字符串视为未设置，回退到 `.default()` 值。`OPENAI_API_KEY` 无默认值，缺失时服务端启动即报错。

**使用位置**: `lib/llm.ts`、`lib/embedder.ts` 均通过 `import { env } from "./env"` 引用。

---

### 2.2 `lib/llm.ts` — LLM 调用封装

**移植自**: `packages/engine/src/llm/index.ts`

**变更**: agentset 通过 Azure OpenAI 调用，映射 `gpt-4.1`/`gpt-5`/`gpt-5.1` 等模型名到 Azure deployment ID。我们使用 `@ai-sdk/openai` 的 `createOpenAI()`，兼容任何 OpenAI 兼容 API（支持自定义 `OPENAI_BASE_URL`）。

```typescript
llmGenerate({ system?, prompt, temperature?, maxTokens? }) → Promise<string>
```

**缺省策略**:
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `temperature` | `0.3` | 偏低以保证知识提取和问答的一致性 |
| `maxTokens` | `4096` | 足够覆盖多片段 RAG 上下文 + 回答 |

**使用位置**:
- `app/api/knowledge/extract/route.ts` — 逐章调用 LLM 提取知识点（`temperature: 0.2`）
- `app/api/rag/query/route.ts` — 基于检索上下文生成回答（`temperature: 0.3`）

---

### 2.3 `lib/embedder.ts` — 嵌入模型封装

**移植自**: `packages/engine/src/embedding/index.ts`

**变更**: agentset 支持 5 种 Provider（Azure OpenAI / OpenAI / Voyage / Google / Managed OpenAI），通过 `WrapEmbeddingModel` 类自动为 Voyage 添加 `inputType` 参数。我们仅保留 OpenAI 兼容 API。

```typescript
embedSingle(text) → Promise<number[]>   // 单文本嵌入
embedBatch(texts[]) → Promise<number[][]> // 批量嵌入
```

**缺省策略**: 模型名从 `env.EMBEDDING_MODEL` 读取（默认 `text-embedding-3-small`）。批量嵌入内部由 `embedMany` 处理，一次调用处理多条。

**使用位置**:
- `app/api/rag/query/route.ts` — 两个调用点：
  - 第 30 行: 批量嵌入分块文本（每批 20 条，`Promise.all` 并行）
  - 第 35 行: 嵌入用户查询问题

---

### 2.4 `lib/chunker.ts` — 文档分块

**移植自**: `packages/engine/src/chunk.ts`

**变更**: agentset 的 `makeChunk()` 依赖 LlamaIndex `TextNode` 和 `metadataDictToNode`，输出包含 `_node_content` 序列化字段。我们去掉 LlamaIndex 依赖，输出纯 TypeScript 对象。

```typescript
chunkText(text, metadata, chunkSize?, overlap?) → ChunkResult[]
```

**分块算法**:
1. 滑动窗口，步长 = `chunkSize - overlap`
2. 在窗口尾部 ±100 字范围内寻找最佳断点（优先级: `。` > `！` > `？` > `\n`）
3. 无法找到断点时按固定长度截断

**缺省策略**:
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `chunkSize` | `600` | 每块 600 字符 |
| `overlap` | `80` | 相邻块重叠 80 字符 |

**使用位置**:
- `app/api/rag/query/route.ts` 第 25 行 — 建立索引时对所有章节内容分块

---

### 2.5 `lib/pdf-parser.ts` — PDF 章节解析

**移植自**: `packages/engine/src/partition/index.ts`

**变更**: agentset 调用外部 Partition API 服务（`PARTITION_API_URL`），传入 `url`/`text`+`parse_options`+`chunk_options`，返回分批结果。我们使用 `pdf-parse` 库本地解析 PDF 为纯文本，再用正则提取章节结构。

```typescript
extractChapters(fullText, textbookId) → Chapter[]   // 正则匹配章节标题
estimatePages(text) → number                          // 1500 字/页估算
getTextbookTitle(filename) → string                   // 去扩展名、替换分隔符
```

**章节识别正则**: `/^第[\d一二三四五六七八九十百千]+章\s*[^\n]*/m`
- 匹配: `第一章 绪论` / `第2章 细胞` / `第十二章 免疫`
- 不匹配时降级为单章 `"全文"`

**缺省策略**: 无法识别章节时，整本教材作为一个 Chapter。每页约 1500 字符（中文教材经验估算）。页码按 40 行/页估算。

**使用位置**:
- `app/api/parse/route.ts` 第 46 行 — PDF 解析后调用 `extractChapters()` 和 `getTextbookTitle()`

---

### 2.6 `lib/txt-parser.ts` — TXT/MD 解析

**移植自**: 无直接对应（agentset 全部走 Partition API）

**功能**: 与 `pdf-parser.ts` 使用相同的章节正则，但额外处理换行符规范化（`\r\n` → `\n`，`\r` → `\n`，`\f` → `\n`）。返回 `{ chapters, fullText }`。

**使用位置**:
- `app/api/parse/route.ts` 第 61 行 — TXT/MD 文件上传后调用 `parseTxtContent()`

---

### 2.7 `lib/vector-store.ts` — 内存向量存储

**移植自**: `packages/engine/src/vector-store/query.ts` + `packages/engine/src/vector-store/common/vector-store.ts`

**变更**: 这是变更最大的模块。agentset 使用抽象的 `VectorStore` 类，具体实现为 Pinecone 或 Turbopuffer（云向量数据库），支持语义/关键词/混合三种检索模式，以及重排序。我们实现为内存数组 + 余弦相似度。

```typescript
class MemoryVectorStore {
  upsert(chunks)              // 按 ID 去重插入
  query(vector, topK, id?)    // 余弦相似度检索，可选按 textbookId 过滤
  deleteByTextbookId(id)      // 按教材删除
  clear()                     // 清空
  get size()                  // 当前 chunk 数量
}
```

**核心算法 — 余弦相似度**:
```
cos(θ) = (A·B) / (||A|| × ||B||)
```

**使用位置**:
- `app/api/rag/query/route.ts` 第 22-36 行 — 懒索引构建 + 检索

**存储生命周期**:
| 事件 | 行为 |
|------|------|
| 首次 RAG 查询 | 检查 `store.size === 0`，若为 0 则遍历所有章节 → 分块 → 批量嵌入 → `upsert` |
| 进程重启 | 全部丢失（内存存储） |
| 切换教材 | 当前实现不自动切换索引（`query` 时按 `textbookId` 过滤，但索引是累加的） |

> **迁移预留**: `MemoryVectorStore` 实现了 `upsert`/`query`/`deleteByTextbookId`/`clear` 接口，后续迁移到 ChromaDB 或 FAISS 时只需替换实现类，调用方（`rag/query/route.ts`）无需修改。

## 3. 数据存储方案

V0.1 全部使用**进程内存存储**，无持久化。

### 3.1 存储清单

| 存储 | 类型 | 定义位置 | 访问方 |
|------|------|----------|--------|
| `textbookStore` | `Map<string, Textbook>` | `app/api/parse/route.ts` (export) | `textbooks/route.ts`、`textbooks/[id]/route.ts`、`knowledge/extract/route.ts`、`rag/query/route.ts` |
| `knowledgeGraphStore` | `Map<string, KnowledgeGraph>` | `app/api/knowledge/extract/route.ts` (export) | `knowledge/[textbookId]/route.ts` |
| `globalVectorStore` | `MemoryVectorStore` (单例) | `lib/vector-store.ts` (模块级 const) | `rag/query/route.ts` |

### 3.2 跨模块共享方式

采用 **模块级单例导出** 模式：API route 文件直接 `export const store = new Map()`，其他 route 通过 `import { store } from "@/app/api/xxx/route"` 直接引用。这是 Next.js 中模块缓存的特性 — 同一进程内 `import` 只执行一次，天然单例。

### 3.3 生命周期

```
用户上传 → textbookStore.set()
用户提取知识 → knowledgeGraphStore.set()
用户首次 RAG 查询 → MemoryVectorStore.upsert() (懒构建)
进程重启 → 全部清空
```

> **已知局限**: `MemoryVectorStore` 索引是累加的 — 上传教材 A → 构建索引 → 上传教材 B 后，B 的向量不会自动加入索引，因为 `store.size !== 0` 跳过了重建。当前解决：刷新页面后重新查询。

## 4. API Routes 缺省与错误处理

### 4.1 统一错误响应格式

所有 API 返回 JSON 格式: `{ error: string }` 或正常数据。HTTP 状态码: 400（参数错误）、404（资源未找到）、500（服务端异常）。

### 4.2 各端点缺省行为

| 端点 | 缺省/降级策略 |
|------|--------------|
| `POST /api/parse` | 不支持的文件格式 → `status: "error"` 而非拒绝整个请求；PDF 解析失败 → 单文件 error，其他文件继续 |
| `GET /api/textbooks` | 无教材时返回空数组 `[]` |
| `GET /api/textbooks/:id` | ID 不存在 → 404 |
| `POST /api/knowledge/extract` | 无章节数据 → 400；LLM 返回非 JSON → 跳过该章继续；Zod 校验失败 → 跳过该章 |
| `GET /api/knowledge/:textbookId` | 未提取知识 → 404 |
| `POST /api/rag/query` | 无检索结果 → 返回 `"当前知识库中未找到相关信息"`（200 OK）；教材未找到 → 404 |

### 4.3 LLM 调用容错

`/api/knowledge/extract` 对每章独立调用 LLM，一章失败不影响其他章。JSON 解析失败或 Zod 校验失败时 `catch { continue }`，不阻断整体流程。最终可能返回部分章节的提取结果。

## 6. 与 agentset 关键差异总结

| 维度 | agentset | V0.1 移植版 |
|------|----------|------------|
| 向量存储 | Pinecone / Turbopuffer 云服务 | `MemoryVectorStore` 内存数组 |
| 文档解析 | 外部 Partition API 微服务 | `pdf-parse` 本地解析 |
| 嵌入 Provider | 5 种 (Azure/OpenAI/Voyage/Google/Managed) | 仅 OpenAI 兼容 API |
| LLM Provider | Azure OpenAI (多模型映射) | OpenAI 兼容 API |
| 重排序 | Cohere / Zeroentropy | 不做 |
| 数据库 | Drizzle ORM + Planetscale | 无（进程内存 Map） |
| 分块 | 可配置 chunk_size/delimiter/language_code | 固定 600 字/80 重叠 |
| 检索模式 | semantic / keyword / hybrid | 仅 semantic (余弦相似度) |
| 类型依赖 | `@llamaindex/core/schema` | 无外部类型依赖 |

## 7. 后续迁移路径

| 模块 | V0.1 方案 | V0.2+ 目标 | 迁移影响范围 |
|------|----------|-----------|-------------|
| 向量存储 | `MemoryVectorStore` | ChromaDB / FAISS | 仅 `lib/vector-store.ts`，接口不变 |
